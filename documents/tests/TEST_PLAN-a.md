# План тестирования (Test Plan)

## 1. Введение
Этот документ описывает стратегию тестирования приложения «Продвинутый менеджер библиотеки музыки» (версия 0.1).
Цель тестирования — верификация функциональных требований (SRS раздел 3.1) и проверка стабильности архитектурного ядра приложения.

## 2. Объект тестирования (Test Items)
Тестируется исполняемый файл приложения `audio-lib-super`, собранный с использованием Qt 6.
**Ключевые компоненты:**
*   Подсистема Ядра (Core) — управление данными и состоянием.
*   Модуль "Библиотека" — сканирование и хранение метаданных.
*   Модуль "Плеер" — воспроизведение и управление очередью.
*   Модуль "Настройки" — управление конфигурацией.

## 3. Риски (Risk Issues)
*   **R-1 (Кодеки):** `Qt Multimedia` зависит от системных кодеков (GStreamer/FFmpeg). На чистой системе воспроизведение может не работать.
*   **R-2 (Файловая система):** Отсутствие прав доступа к выбранным папкам-источникам.
*   **R-3 (Метаданные):** Некорректная кодировка в тегах старых MP3-файлов может вызвать ошибки отображения ("крокозябры").

## 4. Аспекты тестирования (Features to be Tested)
Согласно SRS, тестированию подлежат следующие функции:

### 4.1 Функциональные требования
*   **F-ARCH-1 (Модульность):** Загрузка списка модулей, отображение их статуса.
*   **F-LIB-1 (Управление источниками):** Добавление/удаление папок, сохранение списка между сессиями.
*   **F-LIB-4 (Отображение):** Вывод списка треков в таблицу с метаданными (Артист, Альбом, Год).
*   **F-FILTER-1 (Фильтрация):** Живой поиск по таблице.
*   **F-PLAY-1/2 (Воспроизведение):** Старт/Стоп, навигация, управление очередью (списком воспроизведения).

### 4.2 Нефункциональные требования
*   **NF-REL-3 (Сохранение состояния):** Приложение должно запоминать геометрию окна и положение разделителей.
*   **NF-USE (Удобство):** Корректность отображения UI элементов (активность кнопок).

## 5. Подходы к тестированию (Test Approach)
Будет использовано **ручное системное тестирование методом "черного ящика"**.
*   **Позитивное тестирование:** Проверка основных сценариев использования (Happy Path).
*   **Негативное тестирование:** Попытка использования отключенных функций (например, кнопок недоступных модулей).

## 6. Критерии приемки (Pass/Fail Criteria)
*   **Pass (Успех):** Фактическое поведение системы совпадает с ожидаемым, приложение не падает, данные сохраняются.
*   **Fail (Провал):** Аварийное завершение работы (`SIGSEGV`), потеря данных (источников) после перезапуска, невозможность воспроизведения поддерживаемого формата.
